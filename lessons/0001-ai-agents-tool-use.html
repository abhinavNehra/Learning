AI Agents & Tool Use

From text generation to real-world action — the agent loop explained.

⏱ 75 min 🛠 Hands-on Python

The Problem with Static Prompts

Prompt engineering makes the model think better. Agents make it do things. A model locked inside a context window can only generate text — it cannot check today's stock price, query your database, or send an email. An agent changes that. By giving the model a set of tools it can request, you turn a text generator into a system that can act on the world. That is the leap from prompt engineering to AI engineering.

What you already know
You know how to call the API, craft system prompts, and use chain-of-thought. This lesson builds directly on that — the only new idea is the loop.

The Agent Loop

Every AI agent — no matter how complex — runs this loop:

User Input
LLM
Tool call?
or done?
tool_use
Execute Tool
result sent back
Back to LLM ↺
end_turn
Final Answer
✓ Done

The loop keeps running as long as the model returns stop_reason: "tool_use". Once it returns stop_reason: "end_turn", your loop exits and you return the answer. That's the entire agent pattern — nothing more.

Tool Calling in Python — Real Code

Here is a complete, runnable agent. It has three tools, runs the loop, and prints exactly which tool was called at each step. Paste it into your project and run it.

agent.py — complete runnable example
import anthropic
import json

client = anthropic.Anthropic()

# ── Tool implementations ───────────────────────────────────────────

def get_current_weather(city: str) -> str:
    # Real impl: call a weather API here
    return json.dumps({"city": city, "temp_c": 18, "condition": "Partly cloudy"})

def calculate(expression: str) -> str:
    try:
        result = eval(expression, {"__builtins__": {}}, {})  # safe for numeric expressions
        return str(result)
    except Exception as e:
        return f"Error: {e}"

def search_web(query: str) -> str:
    # Real impl: call Serper, Tavily, or Brave Search API here
    return json.dumps([
        {"title": f"Search result for '{query}'", "snippet": "Stub result — wire up a real search API."}
    ])

# ── Tool registry (maps name → function) ──────────────────────────

TOOL_MAP = {
    "get_current_weather": get_current_weather,
    "calculate":           calculate,
    "search_web":          search_web,
}

# ── Tool definitions (what the model sees) ─────────────────────────
# These descriptions ARE prompts — write them carefully.

TOOLS = [
    {
        "name": "get_current_weather",
        "description": "Get the current weather for a city. Use when the user asks about weather conditions.",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "City name, e.g. 'London'"}
            },
            "required": ["city"]
        }
    },
    {
        "name": "calculate",
        "description": "Evaluate a mathematical expression. Use for arithmetic, percentages, or any calculation.",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {"type": "string", "description": "Python numeric expression, e.g. '1000 / 182.5'"}
            },
            "required": ["expression"]
        }
    },
    {
        "name": "search_web",
        "description": "Search the web for recent information. Use when the user asks about current events or facts you may not have.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Search query"}
            },
            "required": ["query"]
        }
    },
]

# ── The agent loop ─────────────────────────────────────────────────

def run_agent(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]

    while True:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=TOOLS,
            messages=messages,
        )

        if response.stop_reason == "end_turn":
            return response.content[0].text

        # stop_reason == "tool_use" — execute each requested tool
        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue

            print(f"→ Calling {block.name}({block.input})")
            result = TOOL_MAP[block.name](**block.input)
            print(f"← Result: {result}\n")

            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result,
            })

        # Append both the assistant turn and the tool results to history
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user",      "content": tool_results})


if __name__ == "__main__":
    answer = run_agent("What's the weather in Tokyo, and how many degrees warmer is that than 10°C?")
    print("Final answer:", answer)
Run it now
Save as agent.py in your project, make sure ANTHROPIC_API_KEY is set, and run python agent.py. You will see it call get_current_weather and then calculate before answering.

Parallel Tool Calls

In the basic agent loop, the model calls one tool per turn. But when the user asks a question that requires two independent pieces of data, a smart model batches both requests into a single response. Instead of returning one tool_use block, it returns two — and your loop collects all of them before making a single round-trip back to the model.

Both Claude and GPT-4 support this natively. The model decides when calls are independent enough to parallelize. Your job is to make sure your handler already iterates over response.content and collects all tool_use blocks before sending results back — which the code in Section 3 already does correctly. That loop over blocks was not accidental.

The practical impact is real: when two tools have no data dependency on each other (the result of tool A is not needed to call tool B), parallelizing them halves the round-trip latency for that turn.

Production Impact
Parallel tool calls cut latency by 40–60% for turns where tools are independent. On a typical research agent that fetches three external sources per query, that is the difference between a 4-second response and a 2-second response.

Try this query against the agent from Section 3:

parallel_demo.py — both weather calls happen in one turn
# Ask a question that requires two independent tool calls
answer = run_agent(
    "What's the weather in Tokyo AND London right now, and which city is warmer?"
)
print(answer)

# What you will see in the terminal:
# → Calling get_current_weather({'city': 'Tokyo'})
# → Calling get_current_weather({'city': 'London'})
# ← Result: {"city": "Tokyo",  "temp_c": 28, ...}
# ← Result: {"city": "London", "temp_c": 14, ...}
# Final answer: Tokyo is warmer at 28°C vs London's 14°C.
#
# Both calls happened in ONE loop iteration, not two.

Notice that the existing for block in response.content loop in Section 3 already handles this: it collects all tool_use blocks into tool_results, then sends them all back in a single user message. If your handler only processed the first block and returned immediately, parallel calls would silently break.

Interview question
"What is the difference between sequential and parallel tool calls in an agent?"

Sequential: the model issues one tool call per loop turn — each result comes back before the next call is decided. Parallel: the model returns multiple tool_use blocks in a single response turn. You execute all of them and return all results before the next model call. Parallel calls require no data dependency between the tools, and they halve round-trips for that turn.

Production Hardening

The agent in Section 3 works perfectly for learning. Before you ship it, four things are missing. Each one has caused a production incident for someone.

Gap What goes wrong without it The fix
Max iterations guard A hallucinating model or a broken tool result can send the agent into an infinite loop, burning tokens and money until the process is killed Raise an exception if iterations >= 10 (tune per use-case)
Token budget check As the messages[] array grows, you eventually hit the model's context limit and get a hard API error mid-conversation Check response.usage.input_tokens each turn; warn or summarize before the limit
Tool error handling An uncaught exception in a tool crashes the whole agent instead of letting the model recover gracefully Catch exceptions in your tool executor and return a tool_result with is_error: True
Tool call logging When something goes wrong in production, you have no trace of which tools were called, what arguments were used, or how long each took Log tool name, input, output, and latency_ms before returning each result

Here is a production-hardened version of the agent loop incorporating all four patterns:

agent_production.py — hardened loop
import anthropic, json, time, logging

logger = logging.getLogger("agent")
client = anthropic.Anthropic()

MAX_ITERATIONS = 10
TOKEN_BUDGET   = 180_000  # warn at 90% of claude-sonnet-4-6's 200k context

class AgentLoopError(Exception): pass

def execute_tool_safe(block) -> dict:
    """Execute a tool_use block. Returns a tool_result dict (with is_error on failure)."""
    t0 = time.perf_counter()
    try:
        result = TOOL_MAP[block.name](**block.input)
        latency_ms = int((time.perf_counter() - t0) * 1000)
        logger.info("tool=%s input=%s latency_ms=%d", block.name, block.input, latency_ms)
        return {
            "type":        "tool_result",
            "tool_use_id": block.id,
            "content":     result,
        }
    except Exception as e:
        error_msg = f"Tool '{block.name}' failed: {e}"
        logger.error(error_msg)
        return {
            "type":        "tool_result",
            "tool_use_id": block.id,
            "content":     error_msg,
            "is_error":    True,     # model sees this and adapts its strategy
        }

def run_agent_production(user_message: str) -> str:
    messages   = [{"role": "user", "content": user_message}]
    iterations = 0

    while True:
        # Guard 1: max iterations ────────────────────────────────────
        if iterations >= MAX_ITERATIONS:
            raise AgentLoopError(f"Max iterations ({MAX_ITERATIONS}) exceeded")
        iterations += 1

        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=TOOLS,
            messages=messages,
        )

        # Guard 2: token budget ──────────────────────────────────────
        if response.usage.input_tokens > TOKEN_BUDGET:
            logger.warning("Approaching context limit: %d tokens", response.usage.input_tokens)

        if response.stop_reason == "end_turn":
            return response.content[0].text

        # Execute all tool calls (Guards 3+4 inside execute_tool_safe) ─
        tool_results = [
            execute_tool_safe(block)
            for block in response.content
            if block.type == "tool_use"
        ]

        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user",      "content": tool_results})
Key insight: is_error keeps the model in the loop
When you return is_error: True, the model receives the error message as context. It can then retry with different arguments, try a fallback tool, or tell the user what failed — all without your code doing anything special. Compare this to crashing the loop: the user sees nothing and you have no trace of what happened.

Why Does This Work?

The model was trained to produce a special structured response when it determines it needs external data. Instead of guessing, it outputs a tool_use block with the tool name and arguments. Your code receives that, executes the function, and sends the result back as a tool_result message. The model then continues reasoning with the real data.

The model never "runs" anything. It never touches your filesystem, network, or runtime directly. It requests. You execute.

The mental model
The model is the brain. Your code is the hands.
The model decides what to do and when. Your code decides how it actually happens. This separation is what makes agents safe — you control every action that touches the real world.

This also explains why tool descriptions matter so much. The model reads the description to decide whether to call the tool. A bad description = wrong tool choice. A great description = the model uses the right tool at the right time. Writing tool descriptions is prompt engineering.

The React Developer's Mental Model

You already understand this pattern — you've been using it for years in React:

Agents React
Agent loop (perceive → think → act → observe → repeat) Render cycle (props/state → render → DOM update → event → re-render)
Tool result returned to model setState() — new data triggers another think cycle
Tool definition (name, description, input_schema) Component prop interface — a contract the model must satisfy to call the tool
stop_reason: "end_turn" Component returning null — rendering is done, nothing more to produce
Multi-tool agent calling 3 tools in sequence A component that fires 3 side effects in one render cycle before committing

The core idea is identical: data flows in, decisions happen, actions go out, new data flows back in. You've been building reactive loops your whole career — the agent loop is the same thing with an LLM as the decision engine.

Agent Architectures — The Landscape

The loop you built is called ReAct. It is one of three architectures that dominate production AI systems. Knowing all three by name, and knowing when to reach for each one, is expected knowledge at any AI engineering interview.

Pattern How it works Use when Avoid when
ReAct
Reason + Act		 Model alternates between reasoning (thinking about what to do) and acting (calling a tool). Observes the result, then reasons again. Loop continues until end_turn. Most real-world tasks. Dynamic situations where the next step depends on what the current tool returns. Tasks where you already know all the steps upfront — Plan-and-Execute is more efficient there.
Plan-and-Execute First call: model produces a full ordered plan (list of steps). Subsequent calls: execute each step in sequence, optionally revising the plan based on results. Complex multi-step research tasks, code refactoring pipelines, document analysis workflows — anywhere the steps are knowable upfront. Dynamic tasks where step N depends entirely on the outcome of step N-1 and cannot be anticipated.
Reflection Agent produces output, then a second call (same or different model) critiques it and suggests improvements. The original agent revises. Repeat until quality threshold is met. Writing, code generation, structured output extraction — any task where quality matters more than speed and you can define a quality criterion. Real-time response requirements. Tasks with no clear quality metric (reflection without a rubric just adds noise).
Which one is this lesson's agent?
ReAct. It is the most common pattern in production and the pattern every AI company starts with. It is simple enough to debug, general enough to handle most tasks, and maps directly to the tool-calling API. Master ReAct first — then Plan-and-Execute and Reflection are incremental additions on top of it.

Memory in Agents

Every API call to Claude is stateless. The model remembers nothing between calls unless you put it in messages[]. "Memory" in agents is entirely your responsibility, and there are four patterns with distinct trade-offs. This is a top-five interview topic.

Memory Type Where data lives Persists across sessions? Best for
In-context The messages[] array passed with every API call No — lost when your process ends Single-session conversations. Everything you have built so far uses this.
External / Vector DB Embeddings stored in a vector database (Pinecone, pgvector, Chroma) Yes Semantic recall — "remember what I told you last week." Requires embed-on-write and retrieve-on-read. Used in RAG pipelines (Lesson 2).
Structured DB Key facts extracted and stored in a relational or document database Yes User preferences, entity state, settings. Extract facts from conversation, inject relevant ones into the system prompt at session start.
Episodic Full session summaries stored and injected as prior context Yes Long-running assistants that need narrative continuity across sessions. Generate a summary at session end, retrieve it at session start.
Practical guidance for your current skill level
Start with in-context memory — it is what you have already built. Add a structured DB (Postgres or SQLite) when you need to persist user preferences across sessions. Add a vector DB when users need semantic recall ("find that thing I mentioned before"). Episodic summaries are the highest-complexity option; reach for them only when context window cost becomes a real concern on long-running assistants.

Interview Spotlight

These are the six most common agent questions at AI engineering interviews. The answers below are calibrated for a senior full-stack developer pivoting to AI — technically precise, grounded in production realities, not textbook recitations.

How to use this section
Read each answer once. Then close the page and try to answer from memory in your own words. The goal is not memorization — it is building a mental model you can reason from in a conversation.

Q — "Explain the agent loop in simple terms."

Model answer: An agent is a perceive-decide-act-observe loop with an LLM as the decision engine. The model receives input (perceive), decides whether to call a tool or answer directly (decide), your code executes the tool (act), and the result goes back to the model as new context (observe). This repeats until the model decides it has enough information to give a final answer. Nothing runs in the model — it only requests. Your code executes.

Q — "How do you prevent an agent from running forever in production?"

Model answer: Three layers. First, a max iterations guard in the loop — if iterations >= N, raise an exception. Second, a token budget check on response.usage.input_tokens each turn to catch runaway context growth before hitting the API hard limit. Third, an HTTP-level timeout on the entire agent invocation in your API handler or task queue — this catches cases where the model is slow rather than looping. All three are necessary because each one catches a different failure mode.

Q — "What is the difference between sequential and parallel tool calls?"

Model answer: Sequential means the model issues one tool call per loop turn — each result comes back before the next call is decided. Parallel means the model returns multiple tool_use blocks in a single response. Your handler executes all of them and returns all results in one message before the next model call. This halves the round-trips for that turn. The requirement is that the tools have no data dependency — you cannot call tool B with the output of tool A in the same parallel batch.

Q — "How would you handle a tool failure in an agent?"

Model answer: Catch the exception in the tool executor and return a tool_result with is_error: True and the error message as content. Never crash the loop. The model receives the error as context and can adapt — it might retry with different arguments, try a fallback tool, or tell the user what failed. The key is that the model must receive a tool_result for every tool_use it requested; omitting one breaks the message history contract and typically causes a 400 error on the next API call.

Q — "What is ReAct and why is it the dominant agent pattern?"

Model answer: ReAct stands for Reason + Act. The model alternates between a reasoning step (thinking about what it needs) and an action step (calling a tool). This alternation is why it handles complex multi-step tasks well — each new piece of information can change the reasoning for what to do next. It is dominant because it maps directly to the tool-calling API (no orchestration framework needed), it is straightforward to debug by reading the message history, and it handles the majority of real-world agent tasks without requiring a separate planning phase or reflection loop.

Q — "How does an agent maintain memory across sessions?"

Model answer: Never by relying on the model itself — there is no model memory between API calls. For same-session recall, the messages[] array is sufficient. For cross-session recall, you have two main options: a structured database (extract key facts and inject them into the system prompt at session start — good for preferences and entity state) or a vector database (embed past interactions and retrieve semantically similar ones — good for "remember what I said about X"). For narrative continuity on long-running assistants, generate a session summary at the end and inject it at the start of the next session.
